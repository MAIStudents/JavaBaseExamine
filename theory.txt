1. Многопоточность, Пуллы потоков, ExecutorService, Executors

Многопоточность
    Потоки используют ресурсы процесса и позволяют выполнять задачи параллельно. В процессе могут создаваться дочерние потоки, которые 
    могут обмениваться данными между собой.
    Основной поток выполняет main, по завершении которого завершает работу приложение. Для фоновых задач поток можно запустить как 
    daemon, который будет принудительно завершен при окончании работы всех не-daemon потоков процесса.

    Основной поток может создавать другие потоки с помощью интерфейсов Callable или Runnable. Runnable < Callable, т.к. последний может
    пробрасывать исключения и возвращать значения

    В Java поток представлен классом Thread. Создать и запустить поток можно двумя способами:

        1.  Создать подкласс от Thread и переопределить метод run().
            ```
                public class A extends Thread {
                    public void run() {
                        ....
                    }
                    System.out.println(sum);
                }   
                A thread = new A();

            ```
        2.  Реализовать интерфейс Runnable и передать объект полученного класса в конструктор класса Thread.
            ```
                Runnable r = new MyRunnable() { () ->
                    System.out.println(“I'm doomed”);
                }

                Thread t = new Thread(r);
            ```

Пул потоков 
    Контейнер с потоками, выполняющими задачи. За создание и управление пулом потоков отвечают классы и интерфейсы из 
    Executor Framework.

    Объекты, реализующие интерфейс Executor, могут выполнять задачи Runnable. Интерфейс Executor имеет один метод 
    void execute(Runnable v). После вызова этого метода задача будет выполнена асинхронно(в том числе и в дальнейшем)

    ExecutorService наследуется от Executor и добавляет работу с Callable, 
    прерывания выполняемой задачи и завершения работы пула потоков. 
    Для выполнения задач, возвращающих результат, используется метод submit(), который возвращает объект с
    интерфейсом Future<T>. (isDone(), get(), cancel()).
    
    Класс Executors — утилитный класс, создает классы, реализующие интерфейсы Executor и ExecutorService.
    Основные реализации пула потоков:
    
    1. Executors.newCachedThreadPool(): Пул потоков, который создает новый поток, если в пуле не хватает потоков.
    2. Executors.newSingleThreadExecutor(): Пул потоков с одним потоком.
    3. Executors.newFixedThreadPool(): Пул потоков с фиксированным количеством потоков.
    4. ScheduledThreadPoolExecutor: Пул потоков, позволяющий запускать задачи с конкретной частотой.
    

2. Библиотека java.util.concurrent.*: Коллекции для работы с многопоточностью
    Библиотека java.util.concurrent.* предоставляет множество гибких механизмов для работы в многопоточных приложениях.

1. Concurrent Collections
    Обычные коллекции, интерфейсов Set, Map, etc, нельзя использовать в многопоточных приложениях, т.к. никакой синфронизации они 
    не обеспечивают. Поэтому данная библиотека предоставляет разработчику коллекции со схожими интерфейсами, но уже гарантирующие
    безопасную работу нескольких потоков на одной коллекции.

    CopyOnWriteArrayList: Потокобезопасный аналог ArrayList, реализующий алгоритм CopyOnWrite. 
    Он содержит изменяемую ссылку на неизменяемый массив, обеспечивая потокобезопасность без блокировок. 
    Сам алгоритм обеспечивает разделенное владение и создают копию только при выполнении услового set, add, remove etc
    Данный алгоритм гарантирует, что итераторы вернут состояние списка на момент создания самого итератора 
    и не вызовут ConcurrentModificationException    (st cnccrnt excptn).

    ConcurrentHashMap<K, V>: Реализует интерфейс java.util.concurrent.ConcurrentMap и отличается от HashMap и Hashtable 
    внутренней структурой хранения пар key-value. ConcurrentHashMap использует несколько сегментов и можно рассматривать 
    как группу HashMap'ов. По умолчанию количество сегментов равно 16. Доступ к данным определяется по сегментам, а не по объекту. 
    Итераторы этого класса фиксируют структуру данных на момент начала их использования.

    Дальше пачка коллекций расширяет CopyOnWriteArrayList или уже известные нам интерфейсы (можно написать, но долго).

2. Queues
    Неблокирующие и блокирующие очереди для работы в многопоточной среде. Неблокирующие очереди 
    сосредоточены на скорости(т.е. минимум блокировок)
        ConcurrentLinkedQueue
        ConcurrentLinkedDeque

    Блокирующие же очереди наоборот - хоть и теряем в скорости, но можем приостанавливать поток тогда, когда ему не с чем работать
    Producer-Consumer etc
        ArrayBlockingQueue
        LinkedBlockingDeque
        SynchronousQueue

3. Synchronizers
    Вспомогательные утилиты для синхронизации потоков, представляющие собой мощное оружие в “параллельных” вычислениях.
    
    Semaphore: Объект синхронизации. Некоторый счетчик, значение которого задано в конструкторе. Ограничивает доступ к определенному 
    ресурсу. +1 при предоставлении-1 при возвращении.

    CountDownLatch: Объект синхронизации. Счетчик, который отсчитывает число (задается при создании) выполненных условий. 
    При 0 поток(-и) возвращаются в работу
    
    CyclicBarrier: Объект синхронизации типа Барьер. Барьерная синхронизация 
    останавливает участника(-ов) в определенном месте в ожидании прихода остальных потоков.
    По достижение нужного числа у барьера работа продолжается. В отличие от CDL можно повторно 
    искользовать

    Exchanger: Объект синхронизации, используемый для двустороннего обмена данными между двумя потоками. 
    Допускается одностороняя передача. Основнйо метод exchange()

    Phaser: Объект синхронизации типа “Барьер”. В отличие от CyclicBarrier, может иметь серию барьеров.

4. Executors

    Future<V>: Интерфейс для получения результатов работы потока. Объект данного типа возвращает
    ExecutorService, который в качестве параметра получает объект типа Callable<V>. Метод get объекта Future блокирует 
    текущий поток (с таймаутом или без) до завершения работы потока Callable.

    FutureTask<V>: Класс-оболочка, базирующаяся на конкретной реализации интерфейса Future. Чтобы создать реализацию 
    класса FutureTask необходим объект Callable. FutureTask представляет удобный механизм для превращения Callable 
    одновременно в Future и Runnable, реализуя оба интерфейса. Имплементация FutureTask может быть передана на выполнение 
    классу, реализующему интерфейс Executor, либо запущена в отдельном потоке, как класс, реализующий интерфейс Runnable.

6. Atomics
    Классы, которые могут поддерживать атомарные операции над примитивами и ссылками.

    Операция является атомарной, если её можно безопасно выполнять при параллельных вычислениях в нескольких потоках, не используя 
    при этом ни блокировок, ни синхронизацию synchronized.

    compareAndSet - безопасность. Ожидаемое значение или бан
