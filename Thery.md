## Билет 1. Файлы
В Java для работы с файлами используются различные классы, среди которых выделяется класс *File*.
File используется для работы с путями файлов и директорий. 
Он не предоставляет методов для чтения или записи данных, 
но он полезен для создания объектов, представляющих файлы и директории, 
и для получения информации о них. Необходимо включить пакет java.io.
```java 
File file = new File("path/to/file.txt");
```
Это пример создания инстанса.

В  случае применительно к работе с файлами и 
вводом-выводом о потоке говорят, как об абстракции, 
которая используется для чтения или записи информации 
(файлов, сокетов, текста консоли и т.д.).
Объект, из которого можно считать данные, называется **потоком ввода**, а объект, 
в который можно записывать данные, -**потоком вывода**.
В основе всех классов, управляющих потоками байтов, находятся два абстрактных класса:
 **InputStream**(представляющий потоки ввода) и **OutputStream**(представляющий потоки вывода)
Но поскольку работать с байтами не очень удобно, 
то для работы с потоками символов были добавлены абстрактные классы 
**Reader**(для чтения потоков символов) и **Writer**(для записи потоков символов).

В Java существуют различные классы и методы для работы с файлами. 
Для чтения данных из файла можно использовать классы FileInputStream и BufferedReader. 
FileInputStream позволяет открыть поток для чтения байтов из файла, а 
BufferedReader считывает текст из символьного потока ввода, буферизируя прочитанные символы. 
Использование буфера призвано увеличить производительность чтения данных из потока.
Пример ниже:
```java
try {
    FileInputStream fileInputStream = new FileInputStream("file.txt");
    BufferedReader bufferedReader = new BufferedReader(new InputStreamReader(fileInputStream));

    String line;
    while ((line = bufferedReader.readLine()) != null) {
        System.out.println(line);
    }

    bufferedReader.close();
    fileInputStream.close();
} catch (IOException e) {
    e.printStackTrace();
}
```
Для записи можно использовать FileOutputStream(Открывает поток для записи в файл) и 
BufferedWriter(записывает текст в поток, буфферизируя символы => меньше обращений к диску). 

## Билет 2. Java Memory Model - основные аспекты. Garbage Collector. Garbage Collector G1. СMS.
Память в Java можно условно разделить на следующие разделы:
1) Heap(основной сегмент памяти, где хранятся объекты. Он делится на два подсегмента:  Old Generation и New Generation. New Generation делится на подсегменты: Eden и два сегмента Survivor. При запуске Java приложения JVM загружает необходимые классы в кучу)
2) Non-Heap(подразделяется на: Permanent Generation и Code Cache. Permanent Generation содержит набор метаинформации о классах, которая может использоваться, например, в Reflection. В Java 8 PermGen заменён на Metaspace - его динамически изменяемый по размеру аналог. Важно упомянуть, что именно здесь живут статические поля. Так же PermGen\Metaspace относится к "Method Area", в то время как "Code Cache" относится к "Native Area")
3) Stack(это память, своего рода "оперативная память метода", работающая по схеме LIFO. Когда вызывается метод, то для него в памяти стека создаётся новый блок, называемый фрэймом. Содержит:
   - Примитивные типы
   - Ссылки на другие объекты Когда метод завершает работу, блок перестаёт использоваться и освобождается. Размер стековой память намного меньше объёма памяти в куче.)

Модель памяти в Java, используемая внутри JVM, делит память на стеки потоков 
(thread stacks) и кучу (heap).
Все потоки, работающие в JVM, имеют свой стек. Стек в свою очередь держит информацию о том, 
какие методы вызвал поток.
Стек потока содержит в себе все локальные переменные, требующиеся для выполнения методов из 
стека потока. Поток может получить доступ только к своему стеку.
Куча содержит все объекты, созданные в приложении, независимо от того, какой поток создал объект
Статические переменные класса также хранятся в куче вместе с определением класса.
К объектам в куче могут обращаться все потоки, которые имеют ссылку на объект.

### Правила Java Memory Model 

1) однопоточные программы исполняются псевдо последовательно. 
2) нет невесть откуда взявшихся значений.
3) остальные события выполняются по порядку, если связаны отношением строгого частичного порядка “выполняется прежде” .

### Serial Garbage Collector
При использовании данного сборщика куча разбивается на четыре региона, 
три из которых относятся к младшему поколению (Eden, Survivor 0 и Survivor 1), а один (Tenured) — к старшему.

Объект начинает свою жизнь в регионе Eden. 
Сюда его помещает JVM в момент создания. Но со временем может оказаться так, 
что места для вновь создаваемого объекта в Eden нет, в таких случаях запускается малая сборка мусора.
Первым делом такая сборка находит и удаляет мертвые объекты из Eden. 
Оставшиеся живые объекты переносятся в пустой регион Survivor. 

Один из двух регионов Survivor всегда пустой, именно он выбирается для переноса объектов из Eden.
После малой сборки регион Eden полностью свободен и может быть использован для 
размещения новых объектов. Но рано или поздно приложение опять займет всю область 
Eden и JVM снова попытается провести малую сборку, на этот раз очищая Eden и 
частично занятый Survivor 0, после чего перенося все выжившие объекты в пустой регион Survivor 1.
